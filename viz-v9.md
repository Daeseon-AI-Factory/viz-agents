## V9 — 비즈니스 KPI 스트림

### Upgrade from V8
V8 = 기술 아키텍처.
V9 = **비즈니스 차원** — 가치사슬, 돈 흐름, KPI 변화. 기술 변경이 비즈니스에 어떻게 영향?

### Why (본인 비전 매핑)
> "비즈니스 시각화" — Claude가 결제 함수 수정하면 "수익 영향", 알림 추가하면 "engagement 영향" 같은 비즈니스 임팩트 즉시 시각.

### Exit criteria
- KPI 보드: 매출, 활성 사용자, 전환율, 평균 응답시간 등
- 코드 변경 → KPI 추정 영향 (LLM 추론)
- 비즈니스 흐름: 사용자 → 가입 → 결제 → 유지 → 추천
- 돈 흐름 화살표 (수익 / 비용)

### 구체 시각화 예시

```
┌─ 비즈니스 KPI ────────────────────────────────┐
│                                               │
│  💰 MRR        $52K   ▲ +2.3%                │
│  👥 활성 사용자  1,240  ▼ -0.8%               │
│  💳 전환율       3.2%  ━ 변화 없음            │
│  ⚡ p95 응답     320ms ▲ +12ms                │
│                                               │
│  ── 가치사슬 ────────────────────────────     │
│  방문 → [가입] → [결제] → [사용] → [재구독]   │
│  10K     1K       200      150      120       │
│           ↑ Claude 여기 작업 → 가입율 +?      │
│                                               │
└───────────────────────────────────────────────┘
```

### Architecture
- 프로젝트의 비즈니스 메타 (있으면): pricing, plans, KPI 정의
- LLM: 코드 변경 → 비즈니스 임팩트 추정
- 외부 데이터 (선택): Stripe, GA, Mixpanel 등 API 연동
- 클라이언트: KPI 카드 + 가치사슬 시각

### Dependencies
- LLM: 비즈니스 추론
- 외부 API: 선택 (Stripe 등)
- frontend: 차트 (D3, Chart.js)

### Risk
- "비즈니스 임팩트" 추정은 본질적으로 부정확
- 외부 데이터 없으면 가치 제한적
- 프로젝트마다 KPI 다름 → 설정 필요

### V7 와의 차이
- V7 = 사용자가 어떻게 쓰나 (행동)
- V9 = 그 행동이 돈/가치로 어떻게 환산 (결과)
