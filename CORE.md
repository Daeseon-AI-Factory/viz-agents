# VIZ-Core — 제품 핵심 정의

> **이 문서는 viz-core 의 영구 정의서.**
> 모든 기능 추가/제거는 이 정의에 부합하는지로 평가.
> 정의 변경 = 제품 방향 변경 → 신중히.

---

## 한 줄 정의

> **AI 가 발전해도 그 출력을 인간이 즉시 이해하게 시각화하는 R&D 레이어.**

`AI 출력 → viz-core → 사용자의 0.5초 글랜스`

---

## 사용자 페르소나 (절대 잊지 말 것)

- **비-엔지니어** (코드 안 짬)
- **한국어 사용** (영어 약함)
- **집중력 짧음** — 텍스트 wall 못 읽음
- **시각 우선** — 그림 보고 즉시 파악 원함
- **AI 도구 활용자** — Claude Code 등을 매일 씀

→ 이 페르소나의 페인을 직격하지 않는 기능은 **빼야 함.**

---

## 단일 책임 (★ 2026-05-24 명확화)

### DOES (이거만 한다)
1. **시각화 specialist** — 어떻게 더 잘 보여줄지 R&D
2. **모르면 LLM에 물어봄** — Claude API 호출은 "시각 만들기" 목적만
3. **렌더 엔진** — viz_kind 11종 vocabulary + 동적 컴포넌트
4. **글랜스 창** — 항상 켜진 별도 탭

### DOES NOT (이거 절대 X)
- **자체 코드 분석** ← AST 추출 같은 거 X. Claude 가 더 잘함.
- **자체 문서 분석** ← LLM 에 위임
- **자체 비즈니스 로직 추출** ← LLM 에 위임
- 코드 작성 (SI 에이전트 영역)
- 테스트 실행 (QA 에이전트)
- 시스템 운영 (OPS 에이전트)
- 결정 자체
- 영구 데이터 저장 (DB 시스템)
- 인증/권한 관리

### LLM 호출 가이드라인 (★ 매우 중요)
viz-core 의 LLM 호출은 **"시각화 specialist agent"** 역할로만:

```
허용: "이 입력을 viz_kind+viz_data 로 변환해주세요"
허용: "이 repo 구조를 시각화하려면 어떤 데이터가 필요하나요"
허용: "이 코드 의미를 한 줄 + 시각으로 표현하세요"

금지: "이 코드의 버그 찾아주세요" (분석)
금지: "이 함수 리팩터링 해주세요" (생성)
금지: "이 비즈니스 로직 평가하세요" (판단)
```

→ viz-core 는 **"시각화 만들기" 만 LLM에 시킴**. 다른 도메인 X.

### 핵심 메타포

```
   분석/생성/추론        (Claude/외부 AI 책임)
        ↓
   viz-core (시각 specialist)   ← 받아서 잘 보여주기만
        ↓
   글랜스 창
        ↓
   사용자 (즉시 이해)
```

viz-core 의 영역 = **"받은 거를 어떻게 그릴까"** 만.

## 입력 contract

3가지 입력 채널만:

### 1. Hook 이벤트 (자동, default)
```
~/.claude/settings.json hooks
   ↓ Claude Code 도구 호출
   ↓ POST /event
{
  "hook_event_name": "PostToolUse",
  "session_id": "...",
  "tool_name": "Edit",
  "tool_input": { file_path, old_string, new_string }
}
```

### 2. 자연어 요청 (사용자 직접)
```
POST /viz/request
{ "request": "지난 5분 활동을 KPI로 보여줘" }
```

### 3. 외부 에이전트 직접 POST (수동 모드, 미래)
```
POST /event
{
  "hook_event_name": "_summary",
  "viz_kind": "kpi",
  "viz_data": {...}
}
```

이외 입력은 받지 않음.

## 출력 vocabulary (11종 viz_kind)

| viz_kind | 언제 | 예시 |
|---|---|---|
| `diff` | 코드/텍스트 변경 | Edit 의 before/after 빨강/초록 |
| `gauge` | 단일 수치 | 테스트 12/15 통과 |
| `table` | 여러 항목 비교 | 파일 목록, 검색 결과 |
| `flow` | 순서 단계 | 배포 / 마이그레이션 |
| `badge` | 상태 라벨 | 성공/실패/경고 |
| `code` | 코드 스니펫 | 짧은 코드 강조 |
| `animation` ★ | 데이터 흐름 | SVG 점이 노드 사이 흐름 (비전 핵심) |
| `kpi` | 비즈니스 지표 | MRR/사용자/전환율 카드 |
| `journey` | 사용자 여정 | 가입→결제→사용 단계 |
| `arch` | 시스템 컴포넌트 | Frontend/API/DB 레이어 |
| `whatif` | 변경 영향 예측 | 현재 vs 변경 후 비교 |

이게 viz-core 의 **vocabulary**. 외부 에이전트도 이 형식으로 보냄.

확장 정책: 본인이 매일 사용하면서 **빈번한 미매칭 케이스** 발견 시 새 viz_kind 추가. 그 외 무작정 추가 X.

## 핵심 사용 시나리오 (4가지)

### A. 코드 시각화 (가장 흔함)
```
Claude Code 가 Edit api.py
  → hook 자동 → viz-core
  → viz_kind: "diff" 자동 선택
  → 빨강/초록 패치 카드
사용자: 텍스트 안 읽고 변경 즉시 파악
```

### B. AI가 옵션 줄 때
```
Claude 가 AskUserQuestion (옵션 3개)
  → hook 자동 → viz-core
  → 옵션 카드 그리드 (각 카드 = label/description/preview ASCII)
  → 추천 옵션 초록 강조
사용자: 텍스트 wall 안 보고 카드로 비교 선택
```

### C. 작업 흐름
```
사용자 메시지 → Read → Grep → Edit → Bash → Stop
  → 세션 박스 안 ▼ 화살표로 시각
  → Stop 시 LLM 자동 요약 + viz_kind 자동
  → 한 줄 한국어 + 시각
사용자: 0.5초 글랜스로 "이 작업 다 함" 확인
```

### D. 사용자 직접 요청
```
"지난 5분 활동을 KPI로 보여줘"
  → POST /viz/request → LLM → viz_kind + viz_data
  → 즉시 렌더
사용자: 원하는 형식으로 자유 시각
```

## 평가 기준 (모든 기능에 적용)

새 기능 추가 / 기존 기능 유지 결정 시 4문항:

1. **본인 페르소나에 매일 가치 있나?**
   - 비-엔지니어 / 한국어 / 저집중 사용자 페인 직격?
2. **단일 책임에 맞나?**
   - 시각화 외 책임 추가하지 않나?
3. **인지부하 ↑↑ 안 되나?**
   - UI 중구난방 만들지 않나?
4. **AI 발전과 무관한 영구 가치인가?**
   - Claude Code 가 좋아져도 이 기능 가치 유지?

4개 다 ✓ → 추가/유지.
하나라도 ✗ → 빼거나 정리.

## viz-core 의 위치

```
[Claude Code]                 [다른 AI]              [도메인 에이전트]
   ↓ hook                        ↓ POST                  ↓ POST
   └─────────── viz-core (렌더 엔진) ────────────────────┘
                       ↓
                  [브라우저 글랜스]
                       ↓
                    [사용자]
```

viz-core 는 **AI 와 사용자 사이의 시각 다리.**
플랫폼 (ddalkkak.daeseon.ai) 안에 임베드 가능 (iframe / subpath).

## 비-목표 (의식적으로 안 하는 것)

- ❌ Claude/GPT/Gemini 와 경쟁 → 협력 레이어
- ❌ 종합 대시보드 → 시각화 R&D 만
- ❌ Production-grade scaling → 본인 머신 localhost
- ❌ 다중 사용자 권한 → 단일 사용자
- ❌ 모든 viz_kind 무한 확장 → 11종 핵심에 집중
- ❌ AI 자체 성능 향상 → AI 출력 가공
