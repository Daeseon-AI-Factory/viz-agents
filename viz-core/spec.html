viz-core spec

AI 출력을 보지 않고 옆 창만 봐도 이해되도록 — 다른 AI 가 viz-core 가 알아듣는 형식으로 출력하게 하는 명세.

메인 응답은 그대로. viz-spec 코드펜스가 사이드카로 자동 추출됨. LLM 호출 0.

한 그림으로

┌─── 다른 AI (Claude / ChatGPT / Cursor / ...) ───┐ │ │ │ 답변: "회원가입 흐름은 이렇게..." │ ← 사용자에게 보임 │ │ │ ```viz-spec │ ← 사이드카 (안 보임) │ {"viz_kind":"userflow", "data":{...}} │ │ ``` │ └─────────────────┬────────────────────────────────┘ │ 마커만 추출 (LLM 호출 X) ▼ viz-core (localhost:8765) │ ▼ 브라우저 옆 탭 시각화

🔥 복붙 프롬프트

본인 AI 의 system prompt / Custom Instructions 에 박으세요. 한 번만 박으면 끝.

한국어 (짧은 버전)

viz-core 와 페어로 일합니다. 답변이 그림으로 더 명료해질 때, 답변 끝에 viz-spec 코드펜스를 추가하세요. 메인 답변은 그대로 두고, viz-core 가 마커만 뽑아 옆 창에 그립니다. 형식: ```viz-spec {"viz_kind": "...", "data": {...}} ``` 선택 가능한 viz_kind (24종): concept, userflow, screenmap, depgraph, crud, callgraph, arch, timeseries, bar, funnel, heatmap, kanban, waterfall, cohort, kpi, gauge, diff, table, flow, badge, code, animation, journey, whatif, mermaid ★ 자산화 모드 (library 자동 영구 저장): 사용자가 "박아줘" "저장해줘" "등록해줘" 등 영구 저장 의도면 save:true + slug 추가: ```viz-spec { "viz_kind": "concept", "save": true, "slug": "rate-limit", "title": "Rate Limit", "tagline": "한 줄 핵심", "category": "성능", "tags": ["..."], "data": { ...viz_kind 별 schema... } } ``` viz-core 가 library/{viz_kind}/{slug}.json 으로 영구 저장 + 옆 탭 즉시 표시. 규칙: - viz-spec 블록은 사이드카. 메인 답변에서 절대 언급 X. - 한 답변에 1개면 충분. - 애매하면 빼는 게 나음. - 실제 데이터 / 짧게. - save:true 는 사용자가 명시적 자산화 의도 표시 시만. - 각 viz_kind schema: http://localhost:8765/spec

English (full)

You are paired with viz-core, an AI-output visualization layer. When your answer would benefit from a diagram, append a viz-spec code fence at the end. The main answer is unchanged; viz-core picks the fence up and renders it in a side window. Format: ```viz-spec {"viz_kind": "", "data": {...}} ``` Choose viz_kind based on intent: - userflow: user journey with branches (decision nodes) - screenmap: screens + transitions (UX wireframes) - depgraph: service dependencies (microservices) - crud: entity × role permissions matrix - callgraph: function call graph in code - arch: system architecture layers - timeseries: trends over time - bar: category comparison - funnel: conversion stages - heatmap: 2D distribution (time × metric) - kanban: TODO/DOING/DONE board - waterfall: distributed trace spans - cohort: retention matrix - kpi: business metrics cards - gauge: single number with max - diff: before/after code - table: rows × columns - flow: linear steps with status - badge: single status label - code: code snippet - animation: data flowing between nodes - journey: simple user journey (no branches) - whatif: current vs. after comparison - mermaid: anything not above (Mermaid syntax in `code`) Rules: - The viz-spec block is sidecar. Never reference it in the main answer. - One block per answer is enough. Multiple OK if truly separate views. - If unsure, omit. Better no viz than wrong viz. - Real numbers, not placeholders. - Schema per kind: http://localhost:8765/spec

23종 카탈로그

각 viz_kind 의 최소 schema. 실제 예시는 /showcase 에서.

API

POST /viz/spec — 텍스트 받아 viz-spec 추출 + 브로드캐스트

curl -X POST http://localhost:8765/viz/spec \ -H 'Content-Type: application/json' \ -d '{ "text": "...AI 응답 전문...", "source": "claude-code", "session_id": "optional" }' 응답: {"ok": true, "count": N, "specs": [{"viz_kind":"..."}], "llm_used": false}

POST /viz/request — 자연어 → LLM 위임 → viz (예전 방식)

viz-spec 마커 없을 때 폴백. LLM 호출됨. 비용/지연 있음.

curl -X POST http://localhost:8765/viz/request \ -H 'Content-Type: application/json' \ -d '{"request": "월별 매출 추세 그려줘"}'
